# Quality Bar

## Regeln

- **Kein Tutorial-Code** — produktionsreife Patterns auch in MVP
- **Keine halben Features** — Definition of Done erfüllen oder nicht mergen
- **Keine Dummy-Buttons** ohne Backend
- **Keine Fake-Konfigurierbarkeit** — Settings müssen wirken und funktionieren
- **Keine schlechten Settings** — gleiche Qualität wie Core Screens
- **Keine Wegwerf-UI** — funktionale Shell ok, aber mit UX-Brief planen
- Professionelle, **langlebige Architektur** bevorzugen
- Checkliste: `docs/definitions-of-done.md`

## Nicht tun

- „TODO: implement later" als akzeptierter Zustand
- Mock-Dashboards mit Fake-Daten als „Feature fertig"
- Quick hacks die metadata-driven Architektur untergraben
- Scope einer Roadmap-Phase überspringen
